# Video Analyzer V2.2: Performance & Architecture

ご指摘いただいた点を含め、システムの詳細な性能分析と対策をまとめました。

## 1. シーク（ジャンプ）の仕組みと遅延の原因
「解析が終わればバックエンドは不要ではないか」というご指摘は**完全に正解**です。解析後のシーク操作にバックエンド（OpenCV）は一切関与していません。

では、なぜ「10秒の遅延」が発生していたのか？ 2つのフロントエンド側のボトルネックを特定し、修正しました。

### 原因A: React の再描画オーバーヘッド (V2.1 で解消)
- **問題**: 以前の更新で、動画の「現在の再生秒数」をアプリ全体のトップ階層（App）で管理していました。
- **結果**: 動画が動くたび、1秒間に60回、画面上の全てのコンポーネント（ブックマークリスト、タイムライン等）を計算し直していました。ブックマークの数が増えるとこの計算が重くなり、メインスレッドが数秒間フリーズしていました。
- **対策**: 動画プレイヤーを他の UI から完全に絶縁しました。現在はビデオが動いても App 全体は再描画されません。

### 原因B: ブラウザ・デコーダーへの命令過多 (V2.2 で解消)
- **問題**: 今回のような 500MB を超える高解像度動画をシークする際、シークバーを「ドラッグ」すると、ブラウザに対して数ミリ秒おきに数千回の「この時刻を表示せよ」という命令が飛びます。
- **結果**: ブラウザ内蔵のビデオデコーダーが命令を処理しきれず、内部でハング（待機状態）が発生していました。
- **対策**: シーク命令を「100ミリ秒に1回」に制限（スロットリング）しました。これによりブラウザが余裕を持って描画を追従できるようになりました。

## 2. 現在のステータス
